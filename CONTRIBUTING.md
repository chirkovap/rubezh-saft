# Руководство для участников разработки САФТ "Рубеж"

Программный комплекс САФТ "Рубеж" является открытым проектом. Любой вклад в его развитие приветствуется: исправление ошибок, реализация новых функций, улучшение документации, разработка тестов.


## Сообщение об ошибках

При обнаружении ошибки:

1. Проверьте, не зафиксирована ли она уже в разделе [Issues](https://github.com/chirkovap/rubezh-saft/issues).
2. Создайте новое обращение, указав:
   - версию операционной системы и ядра Linux (`uname -a`);
   - шаги для воспроизведения проблемы;
   - ожидаемое поведение;
   - фактическое поведение;
   - вывод журнала (`sudo journalctl -u rubezh-saft -n 50`).


## Предложения по развитию функциональности

1. Создайте обращение с меткой `enhancement`.
2. Подробно опишите предлагаемую функцию и её назначение.
3. Приведите обоснование целесообразности добавления.
4. По возможности предоставьте примеры использования или прототип реализации.


## Порядок оформления Pull Request

1. Выполните fork репозитория.
2. Создайте отдельную ветку для ваших изменений:
   ```bash
   git checkout -b feature/имя-функции
   ```
3. Внесите изменения и проверьте их согласно разделу "Тестирование" данного руководства.
4. Убедитесь, что код соответствует стандартам, описанным ниже.
5. Создайте коммиты с понятными описаниями:
   ```bash
   git commit -m "feat: добавить поддержку IPv6 в XDP-фильтре"
   ```
6. Отправьте ветку в ваш fork:
   ```bash
   git push origin feature/имя-функции
   ```
7. Создайте Pull Request в основной репозиторий с описанием изменений.


## Стандарты кода

### Python

- Стиль: PEP 8
- Имена переменных, функций, классов — на английском языке
- Комментарии, docstrings и сообщения логгера — на русском языке
- Обязательны type hints для всех новых и изменённых функций
- Минимальная версия: Python 3.10+

Пример оформления функции:

```python
def block_ip(ip_address: str, reason: str = "") -> bool:
    """Добавить IP-адрес в чёрный список BPF-карты.

    Args:
        ip_address: IP-адрес в формате строки (например, '192.168.1.1')
        reason: Причина блокировки для журнала событий

    Returns:
        True при успешной блокировке, False при ошибке
    """
    pass
```

### C (eBPF/XDP)

- Стиль: Linux kernel coding style
- Имена функций и структур — на английском языке
- Комментарии внутри кода — на русском языке
- Все указатели должны проходить проверку границ перед разыменованием
- Вспомогательные функции должны быть объявлены как `static __always_inline`
- Имена BPF-карт не изменять: `config_map`, `blacklist`, `whitelist`, `rate_limit`, `stats_map`

### Формат сообщений коммитов

```
тип: краткое описание (до 50 символов)

Подробное описание изменений при необходимости.
Может содержать несколько абзацев.

- Конкретное изменение 1
- Конкретное изменение 2
```

Допустимые типы:
- `feat` — новая функциональность
- `fix` — исправление ошибки
- `docs` — изменения в документации
- `refactor` — рефакторинг без изменения функциональности
- `test` — добавление или исправление тестов
- `chore` — обновление зависимостей, конфигурации сборки


## Тестирование перед отправкой PR

1. Проверить компиляцию XDP-программы:
   ```bash
   cd bpf && make
   ```

2. Проверить синтаксис Python-файлов:
   ```bash
   python3 -m py_compile python/*.py
   python3 -m py_compile web/*.py
   ```

3. Запустить тесты (при их наличии):
   ```bash
   pytest tests/
   ```

4. Проверить установку:
   ```bash
   sudo ./scripts/install.sh
   ```

5. Убедиться в работоспособности сервиса:
   ```bash
   sudo systemctl start rubezh-saft
   sudo systemctl status rubezh-saft
   sudo journalctl -u rubezh-saft -n 20
   ```


## Приоритетные направления разработки

Задачи, в которых участие наиболее востребовано:

- Разработка модульных тестов (pytest) для Python-компонентов
- Улучшение алгоритмов обнаружения DDoS-атак в AttackDetector
- Поддержка IPv6 в XDP-фильтре и Python-демоне
- Интеграция метрик с Prometheus и Grafana
- Реализация автоматической разблокировки IP по истечении таймаута
- Geo-IP фильтрация по базе данных геолокации
- Расширенная статистика по типам атак
- Уведомления по электронной почте или через Telegram при обнаружении атак

Документация:
- Руководства по развёртыванию в различных конфигурациях
- Примеры конфигураций для типовых сценариев применения
- Сравнительный анализ с аналогичными решениями

Инфраструктура:
- Настройка CI/CD pipeline (.github/workflows/)
- Docker-образ для тестового развёртывания
- Ansible playbook для автоматизации установки
